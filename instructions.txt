Task 4 — Chunk and embed documents into a vector store
Split each document into chunks of roughly 400 characters with about 50 characters of overlap.
Embed each chunk with your Week 2 model all-MiniLM-L6-v2 (384-dimensional vectors; the dimension must match between ingestion and query).
Store the chunks in a local vector index. Two free options:
ChromaDB — an in-process vector database that persists to a local folder and gives you metadata, upsert, and persistence out of the box. The simpler default.
FAISS (faiss-cpu) — an in-memory similarity-search index; faster and lighter, but you manage persistence and the id→text/metadata mapping yourself.
Add chunks with stable ids of the form filename#index and the filename in metadata. Use upsert so re-running ingestion does not duplicate chunks.
Wire ingestion into the upload pipeline so new documents are embedded automatically.
Index size, for reference: this corpus produces roughly 50–80 chunks; each embedding is 384 floats ≈ 1.5 KB, so the whole index is well under 1 MB (even 10,000 chunks ≈ 15 MB).
FAISS equivalent: normalize your vectors and use faiss.IndexFlatIP (inner product on normalized vectors = cosine), and keep a parallel list mapping each row to its (chunk text, filename).
Done when: A /rag/reindex endpoint reports how many chunks exist, uploading a new document adds its chunks automatically, and re-running reindex does not increase the count.

Task 6 — Add a reranker (cross-encoder) and measure MRR
Your embedding model is a bi-encoder: it turns the question and each chunk into vectors separately, then compares the vectors — fast and scalable, but the question and chunk never actually “read” each other.
A cross-encoder (a reranker) feeds the question and one chunk through a single model together, so they attend to each other, producing a much sharper relevance score — at the cost of running the model once per pair.
Standard pattern: retrieve a wide top-N cheaply with the bi-encoder, then rerank to a small top-k with the cross-encoder. Use cross-encoder/ms-marco-MiniLM-L6-v2: retrieve top-N = 20, rerank to top-k = 4.
Add a second metric, MRR (Mean Reciprocal Rank): for each question, scan your ranked results for the first chunk whose source file is the correct one (rag_questions_sample_key.csv), note its position (1, 2, 3…), and take 1 / position. Average that over all questions. Report recall@4 and MRR both with and without reranking.
Done when: Your script prints recall@4 and MRR with and without the reranker, and reranking does not lower MRR — it should usually raise it.

Task 10 — Evaluation harness: one scorecard
Pull every metric into one place. Write evaluate.py that reads an answer-key CSV (columns: question, answer_file, expected_keyphrase; sample at rag_questions_sample_key.csv) and prints a single scorecard: recall@k, MRR, answer accuracy, and refusal rate.
Answer accuracy is the fraction of questions whose generated answer contains the expected_keyphrase from the answer key. Where retrieval succeeded but the answer was still wrong, print a one-line diagnosis.
How to compute each metric (concept hints, not code):
Loop once over the answer key and split rows into answerable (have a source file) and unanswerable (blank).
recall@k — for each answerable question, run retrieval, collect the filenames of the top-k chunks, and check membership: is answer_file in that set? recall@k = (number of “yes”) ÷ (number of answerable questions).
MRR — for each answerable question, find the 1-based position of the FIRST top-k chunk whose filename equals answer_file; its contribution is 1 / position (or 0 if the file never appears). MRR is the mean of those reciprocals.
Answer accuracy — run the full /ask pipeline, lowercase both sides, and test whether expected_keyphrase is a substring of the generated answer. accuracy = (matches) ÷ (answerable questions). Crude, but the keyphrases were chosen short and unambiguous so it is good enough.
Refusal rate — run /ask on the unanswerable questions and check whether the answer is (or contains) “I don't know.” refusal rate = (correct refusals) ÷ (unanswerable questions).
Diagnosis line — a question is a “retrieval hit but answer miss” when answer_file was in the top-k yet the keyphrase is absent. Print one line; usual causes are the model ignoring the context, the answer phrasing the fact differently, or the right chunk being out-ranked by a distractor.
Have evaluate.py print, alongside the scorecard, which metric is weakest (its lowest score), so the report is self-explanatory — the numbers are the deliverable, not a write-up.
Done when: One command prints the full scorecard (recall@k, MRR, answer accuracy, refusal rate) and flags the weakest metric.

Task 6 - Rerank and measure MRR

You compute MRR twice: once using only the bi-encoder top-k results (no rerank), and once after reranking. Comparing the two shows how much the cross-encoder improves result ordering.
The bi-encoder is fast but approximate. A cross-encoder scores each (question, chunk) pair together and is much more accurate but too slow for the full corpus. Pattern: fetch top 20 with the bi-encoder, re-score those 20 with the cross-encoder, keep the top 4.

Task 10 - Full scorecard
Task 10 runs against the full question set and produces four numbers: recall@k, MRR, answer accuracy, and refusal rate. It also flags the weakest metric and diagnoses cases where retrieval succeeded but the LLM answer still missed the expected keyphrase.
The four metrics:
recall@k - fraction of answerable questions where the correct file is in the top-k retrieved chunks (from Task 5).
MRR - average reciprocal rank of the correct file across answerable questions (from Task 6).
answer_accuracy - fraction of answerable questions where the LLM answer contains the expected keyphrase. This requires the full RAG pipeline (retrieval + LLM) to be running.
refusal_rate - fraction of unanswerable questions (blank answer_file) where the LLM correctly refuses to answer instead of guessing.
